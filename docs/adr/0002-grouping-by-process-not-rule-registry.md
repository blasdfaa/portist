# 0002 — Группировка по процессу/docker, а не реестр правил «номер → группа»

- Статус: Принято
- Дата: 2026-07-03

## Контекст

Ранняя версия `CONTEXT.md` описывала несуществующую конструкцию: `Group Rule`
как чистый предикат «номер порта → группа», расширяемый реестр через DI-токен
`PORT_GROUP_RULES`, файлы `group-rule.ts`/`builtin-rules.ts` и группы
«БД / Разработка / Системные». В коде ничего этого нет и не было.

Реальная группировка (`PortGrouperService`) ключуется по:
1. docker-контейнеру (джойн по опубликованному порту) → группа «Docker» с
   под-группами по compose-проекту / контейнеру;
2. иначе — точному имени процесса (`processName`), одна группа на процесс;
3. иначе (`processName === null`) — остаточная «Прочее».

Номер порта на выбор группы не влияет. `Port Catalog` даёт строке только имя
сервиса и http-вероятность (флаг `http`) — не группу.

Одновременно `PortKind = "database" | "dev" | "http"` в каталоге был
переусложнён: единственный потребитель (`isLikelyHttp`) схлопывал три значения
в булеву http-вероятность, а `database` не читался вовсе (мёртвые данные). `kind`
свёрнут в `http?: boolean`, мёртвые записи удалены.

## Решение

Группировка остаётся по `processName` + docker-контейнеру. Реестр правил
«номер → группа» и DI-токен **не вводятся**: имя процесса — более точный ключ,
чем диапазон/номер порта, и он уже покрывает известные сервисы (postgres, redis,
node…). Каталог поставляет факты о порте, а не предикаты группировки.

`CONTEXT.md` переписан под эту реальность.

## Последствия

- Классификация «по номеру порта» (группы «Базы данных» / «Разработка») —
  сознательно **не** реализована. Это продуктовая фича, а не архитектурный долг:
  добавляла бы неоднозначность приоритета с группировкой по `processName`
  (куда postgres — в «БД» или в «postgres»?).
- Если такая группировка/бейджи по виду порта понадобятся — это отдельная
  продуктовая работа: вернуть `kind` в каталог и решить приоритет с `processName`.
- Этот ADR существует, чтобы будущий architecture review **не пере-предлагал**
  реестр правил `PORT_GROUP_RULES` как «недостающую точку расширения»: его
  отсутствие — выбор, а не упущение.
